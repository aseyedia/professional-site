Refactoring load_and_derivation: Perl → Trino

A decade-old Perl/PostgreSQL derivation pipeline, re-architected on Apache Trino. Measured head-to-head on identical site submissions — same site, same quarterly release, same submission file — so every comparison is like-for-like. The result was not only a large speedup: the two pipelines produced materially different outputs, and the rewrite's were correct where the legacy pipeline's were not.

Note: Site identities are anonymized to respect institutional data governance. The findings, magnitudes, and methodology below are unchanged.

The legacy pipeline measured runtimes in days, not minutes.

Why the Rewrite Is Faster

The legacy pipeline's runtime was a direct consequence of its execution architecture. Anthropometric derivations ran as interpreted Perl inside of a Docker container, itself executed inside a Python 2.7 Docker container, which made per-patient database round trips: each person's height and weight records were fetched individually, the BMI and z-score calculations were performed row by row in memory, and results were written back to PostgreSQL in batches of 1,000 rows. This architecture required one query cycle per patient and one write cycle per batch, repeated across the full patient population at each site.

The rewrite replaces this with a single set-based SQL statement executed natively in Trino against columnar storage. The per-patient query loop and the interpreted row-by-row computation are both eliminated; the derivation logic is expressed once and applied to the full dataset in one execution. This architectural change accounts for the bulk of the runtime reduction.

Part II: What it was producing

Input validity across the three z-score derivations.

The legacy pipeline processed inputs that did not meet validity criteria — unit-mismatched heights, out-of-range weights, measurements for infants too young to score — and published z-scores computed from them. This affected all three z-score types at every site. A valid z-score distribution has mean ≈ 0 and standard deviation ≈ 1.

The rewrite enforces CDC modified-z screening bounds at the input. Held to those same published thresholds, zero rows breached screening in v2 at any of the ten sites — versus roughly 3.7 million breaching rows the legacy pipeline published.

The Biologically Implausible Value (BIV) cutoffs are the pipeline's own modified-z screening thresholds; v2 is zero for Height-Z and Weight-Z at every site because the screening removes exactly these rows. The legacy pipeline never computes a modified-z, so its Height-Z/Weight-Z figures apply the equivalent threshold to the published z-score as a proxy — illustrative, not strictly equivalent. v2 still retains some |z|>10 values by design (CDC flag-don't-delete) where the modified-z remains within bounds — Site H's weight distribution skews high, but every retained row is sub-threshold.

These findings conclusively demonstrate that the derivations refactor improved and optimized long-standing legacy pipelines. Across 10 production hospital sites and three consecutive submission versions, matched-pair results show a median task-runtime speedup of 256× and a 284× wall-clock reduction (1,664.5h legacy to 5.6h LnDv2). Independently, the rewrite's input screening restored all three z-score distributions to valid ranges at every site: standard deviations fell from as high as the billions to approximately 1, Height-Z extreme values (|z| > 10) were eliminated entirely, and no published row breached the pipeline's own screening thresholds at any site.

Methodology: runtimes from Airflow task_instance.duration (execution time, queue wait excluded), matched on site × version × submission. QC metrics from derived measurement tables (measurement_type_concept_id = 45754907), pipeline-authored rows only, DCC v62 across all 10 sites.
